# Directional VSM Architecture Review

Date: March 12, 2026
Status: `reviewed`
Scope: directional virtual shadow maps only

## 1. Purpose

This document records the March 12, 2026 architecture review of Oxygen's
directional VSM implementation against Unreal Engine's VSM implementation.

Reason for the review:

- user validation reports that Oxygen still shows wrong-page / no-shadow
  behavior during camera or light motion
- repeated local fixes improved individual symptoms but did not close the
  underlying architectural gap
- directional VSM implementation work is therefore frozen until the design
  mismatch is understood and a new plan is written against the correct
  contract

This is a code-inspection review only. No new renderer implementation landed as
part of this document.

## 3. Executive Conclusion

The main failure is not one heuristic. The main failure is the authoritative
contract.

Unreal keeps directional VSM continuity at the page-table and clipmap-cache
level. Oxygen keeps continuity at the whole-publication snapshot level.

That difference explains the recurring motion-time failures:

- Oxygen republishes or suppresses whole directional virtual snapshots
- the published metadata is camera-fit and therefore fragile under motion
- shading returns fully lit if no valid page is found in the currently
  published snapshot
- Oxygen also turns motion into a multi-band shadow-quality problem across
  several simultaneously active clip/refinement regimes, which is exactly the
  kind of large-flat-receiver boiling the user reported on ocean-like surfaces
- backend heuristics (`receiver_bootstrap`, `feedback_refinement`,
  `current_frame_reinforcement`, `last_coherent_publish_fallback`) are trying
  to repair a contract that is unstable by construction

Unreal does not solve this by republishing stale view metadata. It solves it by
keeping a stable virtual address space, preserving page continuity across
clipmap motion, encoding fallback in the page-table entry itself, and updating
physical residency in place.

## 4. What Unreal Does Correctly

### 4.1 Fallback is encoded per page-table entry

In Unreal, page translation does not just say "valid or not." It carries
fallback information in the page-table entry.

Evidence:

- `ShadowVirtualToPhysicalUV()` uses `bAnyLODValid` and `LODOffset` from the
  physical-page entry:
  `VirtualShadowMapPageAccessCommon.ush:269-283`
- `VirtualToPhysicalTexel_PageTableLevelOffset()` also checks whether the
  selected page is valid at the current LOD:
  `VirtualShadowMapPageAccessCommon.ush:250-257`
- `GetMappedClipmapId()` searches for a mapped clipmap level at the sample
  location before the directional projection path shades:
  `VirtualShadowMapProjectionDirectional.ush:42-69`

Implication:

- the shader can fall back to coarser valid data from the same published
  virtual shadow state
- continuity is part of the page-table contract, not a late publish hack

### 4.2 Request generation is visible-sample driven on GPU

Unreal marks pages directly from visible samples and keeps coarse marking as a
separate explicit path.

Evidence:

- pixel-driven page requests are written to `OutPageRequestFlags`:
  `VirtualShadowMapPageMarking.usf:27-33`
- directional page dilation is intentionally small and local:
  `VirtualShadowMapPageMarking.usf:366-378`
- coarse pages are generated by a separate `MarkCoarsePages()` path:
  `VirtualShadowMapPageMarking.usf:485-540`

Implication:

- page requests originate from the pixels actually asking for shadowing
- coarse coverage is explicit and bounded, not mixed into a large CPU-side
  bootstrap heuristic

### 4.3 Physical residency is page-centric and GPU-authoritative

Unreal's residency model preserves and remaps physical pages in place.

Evidence:

- `UpdatePhysicalPages()` remaps previous physical pages to the current frame,
  checks invalidation, preserves valid pages, and updates the page table in
  place:
  `VirtualShadowMapPhysicalPageManagement.usf:142-323`
- new allocations only occur for requested unmapped pages:
  `VirtualShadowMapPhysicalPageManagement.usf:348-418`
- the host-side pass graph wires this as dedicated GPU passes:
  `VirtualShadowMapArray.cpp:751-799`

Implication:

- reuse, invalidation, requested-page protection, and page-table writes are all
  part of one page-centric system
- continuity does not depend on staging a separate "old published snapshot"

### 4.4 Directional clipmap motion is handled by panning/remap, not republish

Unreal keeps clipmap motion stable by snapping and page-space remap.

Evidence:

- clipmap levels are snapped in light space with stable per-level relative
  corner offsets:
  `VirtualShadowMapClipmap.cpp:148-175`
- cache reuse accepts motion through page-space offsets and rejects only when
  guardbands are exceeded:
  `VirtualShadowMapCacheManager.cpp:102-142`

Implication:

- moving the camera does not immediately invalidate the whole notion of the
  published shadow state
- clipmap continuity is an address-space/caching problem, not a shading
  publication problem

## 5. Where Oxygen Failed

### 5.1 Wrong authoritative contract: publication snapshots

Oxygen publishes and reuses whole directional VSM snapshots instead of
maintaining continuity at the page-table level.

Evidence:

- the backend caches and republishes:
  - `last_coherent_shadow_instances`
  - `last_coherent_directional_virtual_metadata`
  - `last_coherent_page_table_entries`
  in `VirtualShadowMapBackend.cpp:557-563`
- it later evaluates whether to republish that old snapshot:
  `VirtualShadowMapBackend.cpp:2660-2837`

Why this is wrong:

- directional virtual metadata is camera-fit
- stale metadata can still be internally coherent while being visually wrong
  for the current frame
- this is exactly the failure the user reported: motion-time wrong pages,
  flashing, then stable convergence

### 5.2 Shading contract is too weak

Oxygen's shader contract still bottoms out at "if nothing valid is found,
return fully lit."

Evidence:

- `ComputeVirtualDirectionalShadowVisibility()` returns `1.0` when no active
  valid page is found:
  `ShadowHelpers.hlsli:1045-1095`

Why this is wrong:

- the shader does not have a strong per-entry fallback contract equivalent to
  Unreal's `bAnyLODValid` / `LODOffset`
- continuity depends on the backend publishing the right page table for the
  frame, rather than the page table carrying its own usable fallback

### 5.3 CPU-side heuristics are compensating for an unstable design

Oxygen relies on a large number of CPU-side planning heuristics to synthesize
coverage under motion.

Evidence:

- `coarse_safety`
- `feedback_refinement`
- `receiver_bootstrap`
- `current_frame_reinforcement`
- `use_last_coherent_publish_fallback_recovery`
  are all live parts of `BuildDirectionalVirtualViewState()`:
  `VirtualShadowMapBackend.cpp:2853-3172`

Why this is wrong:

- these are planning heuristics, not authoritative continuity
- each one can reduce a symptom, but the system still fails whenever the
  published snapshot is wrong for the sample being shaded

### 5.4 Oxygen turned shadow quality into a boiling multi-band problem

This was not recorded explicitly enough before. It is a major design failure.

Oxygen currently makes directional VSM quality under motion depend on several
overlapping mechanisms at once:

- selected clip
- finer clip prefetch / blending
- coarser fallback / blending
- coarse backbone
- receiver bootstrap across multiple fine clips
- feedback refinement
- current-frame reinforcement

Evidence:

- the shader blends between selected, finer, and coarser clip visibility in
  `ShadowHelpers.hlsli:1037-1108`
- the backend independently injects coarse backbone, receiver bootstrap,
  feedback refinement, and reinforcement in
  `VirtualShadowMapBackend.cpp:2899-3175`

Why this is wrong:

- on large flat receivers such as ocean/water planes, many samples ride these
  moving clip-quality boundaries at once
- that creates visible boiling or flashing between quality bands during camera
  motion instead of a stable degrade/recover behavior

What Unreal does instead:

- Unreal still has clipmap levels, but the page request and content model is
  much simpler at the page-flag level
- normal requested pages carry one explicit `VSM_DETAIL_GEOMETRY_FLAG`, while
  coarse pages do not:
  `VirtualShadowMapPageAccessCommon.ush:12-22`
  and `VirtualShadowMapPageMarking.usf:81-82`
  / `VirtualShadowMapPageMarking.usf:505-540`
- that is a much simpler binary coarse/detail content policy than Oxygen's
  layered bootstrap/reinforcement quality machinery

### 5.5 Oxygen failed to optimize address space with flags and page-space reuse

This is the second major failure that was only implied before and should be
stated directly.

Unreal optimizes virtual space by carrying compact page flags and by reusing
page space aggressively across frames.

Evidence:

- Unreal page flags explicitly track:
  - allocated
  - dynamic uncached
  - static uncached
  - detail geometry
  - used this frame
  in `VirtualShadowMapPageAccessCommon.ush:12-32`
- Unreal preserves and reuses page-space mappings with page-address offsets and
  previous/current physical page metadata in
  `VirtualShadowMapPhysicalPageManagement.usf:232-299`
- Unreal clipmap cache reuse is based on page-space panning/remap in
  `VirtualShadowMapCacheManager.cpp:137-141`

What Oxygen does instead:

- it keeps introducing new page demand through bootstrap/refinement/
  reinforcement and then tries to budget it
- it does not have an equivalent compact page-flag contract separating coarse
  versus detail geometry participation
- it does not treat clipmap motion primarily as a page-space reuse/remap
  problem

Why this is wrong:

- it wastes address space on churn instead of reusing it
- it increases pressure on exactly the frames where motion is already hardest
- it contributes directly to the wrong-page / boiling symptoms because the
  system is busy replanning quality bands instead of preserving coherent page
  identity

### 5.6 Oxygen's page-table contract is too thin

This is another core design miss.

Oxygen's page-table entry only carries:

- physical tile x
- physical tile y
- valid bit
- requested-this-frame bit

Evidence:

- `kPageTableValidBit` and `kPageTableRequestedThisFrameBit`:
  `VirtualShadowMapBackend.cpp:38-39`
- `PackPageTableEntry()`:
  `VirtualShadowMapBackend.cpp:218-223`

Why this is wrong:

- it carries no per-entry fallback level
- it carries no "any coarser LOD valid" signal
- it carries no content-state distinction comparable to Unreal's page flags

That is why Oxygen keeps having to recover continuity through publication logic
and CPU-side heuristics instead of through the sample contract itself.

### 5.7 Oxygen left the backend monolithic

This was also under-recorded before.

The current backend still centralizes too many responsibilities in two large
functions:

- `BuildDirectionalVirtualViewState()`:
  `VirtualShadowMapBackend.cpp:1897`
- `ResolvePendingPageResidency()`:
  `VirtualShadowMapBackend.cpp:723`

Those functions currently mix:

- clipmap fitting
- compatibility gating
- feedback policy
- bootstrap/refinement policy
- coarse safety policy
- resident-page carry and eviction
- page-table writes
- per-page view-constant construction
- raster schedule building
- fallback publication state
- diagnostics

Why this is wrong:

- it prevents clean ownership boundaries
- it makes it hard to replace one stage without rebreaking several others
- it is the opposite of Unreal's split between page marking, page management,
  cache invalidation, draw-command building, raster, and projection

### 5.8 Oxygen never split invalidation / page management / draw-command build

Unreal has dedicated stages for these concerns.

Evidence:

- hierarchical page flags:
  `VirtualShadowMapArray.cpp:736-748`
- physical page update:
  `VirtualShadowMapArray.cpp:751-799`
- dirty-flag update:
  `VirtualShadowMapArray.cpp:2169-2180`
  and `VirtualShadowMapArray.cpp:3000-3015`
- per-page draw-command build:
  `VirtualShadowMapBuildPerPageDrawCommands.usf:176-418`
  and `VirtualShadowMapArray.cpp:2129-2167`

What Oxygen does instead:

- invalidation, allocation, residency, and raster scheduling are still folded
  into the backend path centered on
  `VirtualShadowMapBackend.cpp:723-1038`

Why this is wrong:

- it prevents page-centric reuse from becoming authoritative
- it keeps raster scheduling CPU-authored instead of page/authored-state driven
- it makes every optimization fight the same monolithic function

### 5.9 Coarse fallback was bolted on too late

Oxygen's coarse fallback exists as a backend policy, not as a first-class
sample contract.

Evidence:

- coarse safety is planned and budgeted in the backend:
  `VirtualShadowMapBackend.cpp:2039-2083`
  and `VirtualShadowMapBackend.cpp:2653-2658`
- the allocator even prioritizes it under pressure:
  `VirtualShadowMapBackend.cpp:950-960`

Why this still failed:

- a mapped coarse budget does not guarantee the receiver sample has a valid
  coarse page in the published state
- the fallback remained a planning goal, not a shading-level guarantee

### 5.10 Resolve ownership improved, but the architecture stayed publication-led

Some earlier work correctly centralized resolve ownership, but it did not
change the deeper problem.

Evidence:

- the resolve path became the owner of current-frame page-table prep and
  raster-plan materialization in Oxygen
- but the system still depends on `last_coherent_publish_fallback` and
  camera-fit metadata reuse when the current state is judged incoherent:
  `VirtualShadowMapBackend.cpp:2958-2969`

Conclusion:

- this was a real cleanup, but it did not fix the central design error

## 6. Correct Architectural Direction

If directional VSM work resumes, the next design must change the contract, not
just the heuristics.

### 6.1 Stable page-table continuity must replace stale-publication fallback

The system should stop using last-published whole snapshots as the primary
motion-time continuity mechanism.

Required change:

- continuity must come from stable virtual addressing plus page-table reuse and
  remap
- page-table entries must carry a usable fallback contract for shading
- stale full-metadata republish should become an exceptional escape hatch at
  most, not the central design

### 6.2 Directional clipmaps must be panned/reused as clipmaps

The directional address space must behave like a panning clipmap cache, not a
fresh camera-fit publication that is later judged "compatible enough."

Required change:

- preserve stable snapped clipmap identity
- remap prior pages by page-space offset when motion stays within reuse bounds
- invalidate only when guardbands or content changes truly require it

### 6.3 Request generation should be predominantly GPU visible-sample driven

Receiver bootstrap and reinforcement should not remain the main way Oxygen
finds pages under motion.

Required change:

- visible-pixel page marking should be authoritative
- coarse requests should be explicit, bounded, and separate
- CPU-side bootstrap should shrink to a narrow recovery mechanism rather than a
  main source of page demand

### 6.4 Coarse fallback must be sample-guaranteed

It is not enough to reserve coarse pages in the allocator.

Required change:

- at least one coarse level used by shading must be guaranteed by design to fit
  and be queryable for the receiver sample
- fallback must be represented in the published page-table contract, not only
  in backend counters and budgets

## 7. Practical Failure Summary

The architecture failed in these concrete ways:

1. Oxygen made the published directional metadata too unstable.
2. Oxygen used whole-snapshot stale republish as a motion continuity tool.
3. Oxygen turned quality under motion into a boiling multi-band problem.
4. Oxygen failed to optimize space through compact page flags and aggressive
   page-space reuse.
5. Oxygen left the page-table contract too thin to carry fallback and content
   state.
6. Oxygen left the backend monolithic instead of splitting page marking, page
   management, invalidation, and per-page draw scheduling.
7. Oxygen left fallback too weak at the shading/page-table contract.
8. Oxygen overused CPU bootstrap heuristics to compensate.
9. Oxygen implemented coarse safety as backend policy instead of sample-level
   guarantee.

That combination is why the implementation could look correct in the stable
state and still fail under motion.

## 8. Status Impact

This review changes the truthful state of the current directional VSM effort:

- functional stable-state validation is still valid
- the current architecture is not sufficient to close motion-time correctness
  under aggressive camera/light changes
- the current performance-recovery plan is frozen because it is optimizing a
  contract that is now known to be architecturally wrong for the remaining
  correctness issue
- the replacement redesign plan now lives in
  `src/Oxygen/Renderer/Docs/directional_vsm_redesign_plan.md`
